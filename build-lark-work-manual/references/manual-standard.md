# Work Manual Standard

Use this reference while drafting or reviewing a work manual.

## Evidence Rules

| Class | Meaning | Writing rule |
| --- | --- | --- |
| Confirmed fact | Directly supported by an authoritative source | State plainly and retain a source link |
| Synthesized practice | Conservative pattern supported by multiple observations | Use wording such as “通常”“主要”“当前可归纳为” |
| Recommended standard | Proposed improvement or missing control | Label it “建议” and do not imply adoption |
| Open question | Material gap or conflict | Put it in “待确认事项” with the decision owner when known |

Prefer official structured sources over chat, newer decisions over obsolete instructions, and actual operating evidence over generic job descriptions. Never copy private conversation content verbatim unless necessary; summarize the operational fact.

## Coverage Checklist

Include sections that carry real information. Merge adjacent sections when the role is small.

1. Document purpose, audience, version, baseline date, maintainer, and evidence note.
2. Role mission, business context, and value chain position.
3. Success outcomes and responsibility boundaries: owns, collaborates, does not own.
4. Responsibility map with inputs, actions, outputs, collaborators, systems, and cadence.
5. Daily, weekly, monthly, quarterly, and annual operating rhythm.
6. End-to-end SOPs for every core responsibility.
7. Decision rights, approvals, escalation paths, and exception handling.
8. Data sources, system access, table ownership, naming conventions, and update rules.
9. Metric dictionary with formula, source, frequency, owner, and interpretation.
10. Meeting map with purpose, participants, inputs, outputs, and follow-up mechanism.
11. Stakeholder map and RACI or equivalent ownership table.
12. Templates, reports, checklists, communication patterns, and reusable links.
13. Risk register, controls, warning signals, and contingency actions.
14. Common scenarios, failure modes, and lessons learned.
15. Onboarding, handover, backup coverage, and minimum viable continuity pack.
16. Improvement backlog, recommended standards, and open questions.
17. Source register and revision history.

## SOP Pattern

For each core workflow, cover:

| Element | Required detail |
| --- | --- |
| Trigger | Event, cadence, threshold, or request that starts work |
| Inputs | Documents, fields, systems, and responsible suppliers |
| Steps | Ordered actions and decision points |
| Outputs | Deliverable, destination, and completion evidence |
| Timing | Deadline, service level, or meeting cutoff when confirmed |
| Controls | Validation, reconciliation, approvals, and audit trail |
| Exceptions | Common failure paths and escalation owner |
| Dependencies | Upstream and downstream collaborators |

## Metric Pattern

Do not invent targets. A useful metric entry includes:

- metric name and business purpose;
- exact formula and unit;
- source system or table;
- refresh frequency and data owner;
- interpretation and known caveats;
- target or threshold only when confirmed, otherwise a clearly labeled recommendation.

## Quality Gate

A manual is ready when a qualified backup can use it to locate inputs, run recurring work, produce outputs, identify owners, recognize exceptions, and escalate correctly without relying on undocumented tribal knowledge.
